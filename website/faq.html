Frequently Asked Questions

Does this replace GitHub Projects or Issues?

No. PlanSync uses Issues and Projects as the collaboration surface. It creates them automatically from the plan, but you continue to use GitHub normally for discussions, status updates, and project management.

What if my team uses different AI tools?

PlanSync generates context files for AGENTS.md (28+ agents), CLAUDE.md, .cursorrules, .windsurfrules, .github/copilot-instructions.md, GEMINI.md, and .continue/rules/00-plansync.md. Each tool gets the format it expects. Adding support for a new tool means dropping a .tmpl file into .plansync/templates/.

Is the scope enforcement secure?

The chmod-based enforcement stops accidental writes outside a task's scope. It is not an adversarial security boundary — the same OS user that owns the file can revert the permission. The GitHub Action scope check is the real backstop for intentional violations.

Can I write a plan manually?

Yes. The plan file (.plansync/plan.json) is a simple JSON document that validates against a Zod schema. Your coding agent can write it directly after reading AGENTS.md — or write it by hand and run plansync delegate.

Does PlanSync send my code anywhere?

No. The source code never leaves your machine or your CI runner. If you use plansync plan with an LLM, only your project description is sent to the API. Without it, no data leaves at all.

What if a collaborator doesn't run plansync sync?

They'll have write access to everything until they run it. The post-merge git hook (installed by plansync init) runs it automatically after every pull. CI will still flag any scope violations in their PRs.

How do I update the plan after delegation?

Edit .plansync/plan.json (or regenerate with plansync plan), then run plansync delegate again. It will create new Issues for new tasks and skip existing ones.
