How PlanSync works

PlanSync is a CLI tool that turns a project plan into GitHub Issues, assigns scope permissions, and generates agent context files. Two common scenarios:

Starting a new project

  1. Create a repo — git init + push to GitHub
  2. npm install -g plansync — install globally
  3. plansync init — authenticate, scaffold workflows, write AGENTS.md
  4. Agent plans the work — your coding agent reads AGENTS.md and writes .plansync/plan.json
  5. plansync delegate — Issues, Project board, scope manifests, commit
  6. plansync sync — locks scope permissions + generates per-user context files at project root (auto-discovered by all AI agents)

Adopting an existing project

  1. cd existing-project — already has a GitHub remote
  2. npm install -g plansync — install globally
  3. plansync init — re-authenticate, adds .plansync/config.json to .gitignore
  4. Agent plans the refactor — describe the next milestone to your agent
  5. plansync delegate — same as new project
  6. plansync sync — locks scope + generates per-user context files at project root (e.g. AGENTS.md) + .plansync/context/<name>/
  7. plansync clean — undo everything if you change your mind

All commands

CommandWhen to useWhat it does
plansync init First time setting up PlanSync on a repo Authenticates with GitHub (browser or PAT), saves credentials to .plansync/config.json, copies CI workflows to .github/workflows/, installs a post-merge git hook, writes AGENTS.md with planning instructions for your coding agent, and adds config to .gitignore.
plansync delegate [--auto] Your plan is ready — agent has written .plansync/plan.json Shows the plan, asks for task assignments (numbered menu: "1=b" or "all=a"), then creates GitHub Issues, a Project v2 board, scope manifests (.plansync/scopes/*.json), and PROJECT_PLAN.md. Commits everything and pushes. Use --auto to skip reassignment and use round-robin defaults. Re-running after plan edits always deduplicates automatically — no special flag needed.
plansync status Any time — check what's happening Prints the plan with live GitHub Issue states. Shows task status, assignee, scope globs, blocking dependency chains, and a summary count.
plansync sync [<username>] [--user <name>] [--supervisor] [--reset] Set up your environment after delegation Auto-detects if you're the repo admin via GitHub API. Admins: all files stay writable; root AGENTS.md gets planning instructions, assigned tasks, and an all-tasks overview table. Collaborators: assigned files become writable (chmod 644), everything else read-only (chmod 444). Per-user context files are written to the project root (e.g. AGENTS.md, CLAUDE.md, .cursorrules) for agent auto-discovery, plus .plansync/context/<name>/ as a backup. Username can be passed as a positional argument, via --user flag, PLANSYNC_USER env var, or auto-detected from GitHub. Use --supervisor to keep all files writable. Use --reset to restore all files to writable.
plansync plan <desc> Optional — if you want an LLM to help draft the plan Sends your project description to an LLM to generate a structured plan. Supports approve, edit with feedback, regenerate, or cancel. Saves to .plansync/plan.json.
plansync whoami Verify your identity Prints your GitHub username and role (admin/collaborator). Resolved from --user flag, PLANSYNC_USER env var, or GitHub token. No token needed — works with --user or env var alone.
plansync clean Undo — remove all PlanSync traces Deletes .plansync/, workflow files, post-merge hook, PROJECT_PLAN.md, and removes <!-- plansync --> blocks from context files. Removes .plansync/config.json from .gitignore.

End-to-end example

$ npm install -g plansync

$ mkdir my-app && cd my-app
$ git init && git add -A && git commit -m "first"
$ gh repo create my-app --push --public
$ plansync init
  # Press Enter → browser opens → authorize → done

  # Your agent reads AGENTS.md and writes .plansync/plan.json:
  #    T001 - "Set up auth"  → scope: src/auth/**, backend/auth/**
  #    T002 - "Build landing" → scope: src/pages/**

$ plansync delegate
  # Tasks: 1. T001  2. T002          Collaborators: a. howards254  b. janedoe
  # Reassign (e.g. "1=b" or "all=a"), or Enter to approve
  # Approve → issues created → manifests committed → pushed

$ plansync status
  # T001   ◻ ready   howards254  src/auth/**, backend/auth/**
  # T002   ◻ ready   janedoe     src/pages/**

$ plansync sync --user janedoe
  # Only src/pages/** is writable for janedoe
  # Context files written to root (AGENTS.md, CLAUDE.md, .cursorrules, ...)
  # Backup copy at .plansync/context/janedoe/

$ plansync sync
  # Admin detected — all files stay writable
  # Root AGENTS.md gets planning instructions + all-tasks overview

Updating plans after delegation

Plans change. When you need to modify the plan after delegation:

  1. Edit .plansync/plan.json — Add, remove, or modify tasks. You can also regenerate it with plansync plan if using the LLM.
  2. Re-delegate — just run plansync delegate again. It automatically matches task IDs against existing Issues and intelligently updates GitHub: 
    $ plansync delegate
  # Fetching existing Issues...
  # ✓ Updated Issue #1 (T001: Set up auth)
  # ✓ Created Issue #4 (T004: Write tests)
  # ✓ Closed Issue #2 (T002: removed from plan)
  # ✓ Committing and pushing...
    New tasks create new Issues, changed tasks update existing Issues, removed tasks close their Issues with a comment explaining removal. No special flag needed — deduplication is always on.
  3. Major rewrites — If you've made sweeping changes (reordered all tasks, changed scope structure), use plansync clean followed by a fresh plansync delegate for a clean slate.

Note: Only repository admins (owners) can run plansync delegate. Collaborators with push access can run plansync sync to lock their scope, but cannot delegate new plans.

Role-based guide

Admin (Owner)

  1. plansync init — authenticate, scaffold workflows and hooks
  2. Agent writes the plan — reads AGENTS.md → writes .plansync/plan.json
  3. plansync delegate — push Issues, Project board, scope manifests to GitHub
  4. plansync sync — auto-detects admin, keeps all files writable, generates rich context
  5. Update plan: edit plan.json → plansync delegate again (dedup is automatic)
  6. plansync whoami — verify active GitHub identity
  7. plansync sync --supervisor — (optional) keep files writable + context only

Collaborator

  1. git pull — post-merge hook runs plansync sync automatically
  2. plansync sync <username> — locks scope permissions, generates per-user context files
  3. plansync status — view your tasks with live Issue states
  4. If sync says "No plan found": ask admin to run plansync delegate first
  5. To write outside scope: plansync sync --reset restores all files to writable
  6. plansync whoami — check which identity you're authenticated as

Supervisor mode for project owners

As the project owner, you may want to oversee all tasks without scope restrictions. Use --supervisor mode:

$ plansync sync --supervisor
  # ✓ All files remain writable (supervisor mode)
  # ✓ Generated context files for howards254
  # ✓ Context files written to .plansync/context/howards254/

This keeps all files writable for you while still generating your context files. You can read and write anywhere in the project, while collaborators remain locked to their assigned scopes.

When to use supervisor mode:

  • You're the project owner reviewing all work
  • You need to make cross-cutting changes (refactoring, bug fixes)
  • You're coordinating between multiple collaborators

When to use regular sync:

  • You're assigned specific tasks and want to stay within your scope
  • You want the same protection as collaborators (preventing accidental writes outside your scope)
