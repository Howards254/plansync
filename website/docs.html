Commands

plansync init

Sets up PlanSync on a repository. Authenticates with GitHub, scaffolds CI workflows and a post-merge hook, writes AGENTS.md for your coding agent, and ensures .plansync/config.json is gitignored (untracks it if previously committed).

$ plansync init
  Enter to open browser → authorize → done
  ✓ Authenticated as your-github-username
  ✓ .plansync/config.json saved and gitignored
  ✓ Workflows installed (.github/workflows/scope-check.yml, ...)
  ✓ Post-merge hook installed (.git/hooks/post-merge)
  ✓ AGENTS.md written

To use a PAT instead of the browser flow, paste the token at the prompt. The token needs repo and project scopes. Generate one at github.com/settings/tokens/new.

plansync delegate [--auto]

Reads .plansync/plan.json (written by your agent) and pushes it to GitHub as Issues, a Project v2 board, and scope manifests. Auto-distributes tasks round-robin to repo collaborators. Shows a numbered reassignment menu so you can move tasks between collaborators quickly before approving.

$ plansync delegate

  Tasks:                              Collaborators:
   1. T001  Set up auth                a. howards254 (you)
   2. T002  Build landing page         b. janedoe
   3. T003  Write API routes           c. alice

  Current assignments:
   1 → a (howards254)
   2 → b (janedoe)
   3 → a (howards254)

  Reassign (e.g. "1=b" or "1,3=b" or "all=a"), or Enter to approve:
  ✓ Creating Issues...
  ✓ Creating Project board...
  ✓ Writing scope manifests...
  ✓ Committing and pushing...

Use --auto to skip the reassignment menu entirely and use round-robin defaults (the approval prompt is still shown). If a username is not a repo collaborator, the issue is created unassigned with a comment noting the intended assignee.

plansync status

Prints the plan alongside live GitHub Issue states. Shows each task's status (ready / in progress / done / blocked), assigned user, scope globs, Issue number, and blocking dependency chains. Ends with a summary count.

$ plansync status

  T001  ◻ ready    howards254  #1  open   src/auth/**
  T002  ◻ ready    janedoe     #2  open   src/pages/**
  T003  ◆ blocked  howards254  #3  open   src/api/**  blocked by T002

  Summary: 3 total · 2 ready · 1 blocked · 0 in progress · 0 done

plansync sync [--user <name>]

Locks filesystem permissions to a collaborator's assigned scope. Files matching the user's task scopes become writable (chmod 644); everything else becomes read-only (chmod 444). Prevents accidental edits outside your scope. Also generates per-user context files at .plansync/context/<username>/ containing only the current user's assigned tasks. Username is resolved via --user flag, PLANSYNC_USER env var, or auto-detected from the GitHub token. Use --reset to restore all files to writable.

$ plansync sync --user janedoe

  ✓ Set writable: src/pages/index.html
  ✓ Set writable: src/pages/about.html
  ✓ Set writable: src/pages/style.css
  ✓ Set read-only: src/auth/login.js
  ✓ Set read-only: src/api/routes.js
  ✓ Set read-only: backend/server.js
  Permission Summary: (2 writable, 3 readonly)

  Generating context files for janedoe...
  Wrote 7 context file(s) to .plansync/context/janedoe/

$ plansync sync --reset
  ✓ All files restored to writable

Skips .git/, node_modules/, and .plansync/ directories automatically. The CI scope-check Action serves as the real backstop; local permissions are an opt-in convenience.

plansync plan <description>

Optionally uses an LLM to draft a plan. Sends your project description to Claude (requires ANTHROPIC_API_KEY) and returns a structured plan with tasks, scopes, dependencies, and acceptance criteria. Supports an interactive review loop — approve, edit with feedback to regenerate, or cancel. Saves the result to .plansync/plan.json.

This command is entirely optional. You can skip it and have your agent write .plansync/plan.json directly by reading AGENTS.md.

$ export ANTHROPIC_API_KEY=sk-ant-xxxxx

$ plansync plan "Build a todo app with React frontend and Express backend"

  Generating plan...
  ┌────────┬──────────────────────┬────────────────────┐
  │ Task   │ Description          │ Scope              │
  ├────────┼──────────────────────┼────────────────────┤

plansync clean

Completely removes PlanSync from a project. Deletes the .plansync/ directory, removes workflow files (scope-check.yml, silence-detection.yml), the post-merge hook, and PROJECT_PLAN.md. Strips all <!-- plansync --> / <!-- end-plansync --> blocks from context files. Removes .plansync/config.json from .gitignore. After running clean, you can start fresh with plansync init.

$ plansync clean

  ✓ Deleted .plansync/
  ✓ Deleted .github/workflows/scope-check.yml
  ✓ Deleted .github/workflows/silence-detection.yml
  ✓ Deleted PROJECT_PLAN.md
  ✓ Stripped plansync markers from AGENTS.md
  ✓ Stripped plansync markers from CLAUDE.md
  ✓ Stripped plansync markers from .cursorrules
  ✓ Removed .plansync/config.json from .gitignore
  PlanSync removed. Run 'plansync init' to start fresh.

Architecture

PlanSync uses four layers:

  1. Plan generation — After plansync init, your coding agent reads AGENTS.md and writes .plansync/plan.json with tasks, scope, and dependencies. You review, revise, approve.
  2. GitHub synchronization — The delegate command writes the plan to GitHub Issues, a Project board, and scope manifests. Assignments are written back to .plansync/plan.json.
  3. Scope enforcement + context — The sync command walks the repo and applies filesystem permissions. Only your assigned files are writable. It also generates per-user context files at .plansync/context/<username>/.
  4. CI monitoring — A GitHub Action checks every PR for scope violations. A scheduled action flags stalled tasks.

Key design decisions:

  • GitHub is the source of truth — plans, manifests, and context files are plain files committed to the repo
  • No server — PlanSync is a CLI tool and a set of CI actions. Nothing to host.
  • No API gate — Scope enforcement uses chmod. Any tool respects it.
  • CI is the backstop — local permissions prevent accidents; the GitHub Action prevents deliberate scope drift.

Context files

When you run plansync sync --user <username>, PlanSync generates per-user context files containing only the tasks assigned to you. Compatible with 28+ coding agents:

FileAgents
AGENTS.mdOpenCode, Codex CLI, Cursor, GitHub Copilot, VS Code AI, Devin, Aider, Zed, Warp, JetBrains Junie, Gemini CLI, Goose, Factory, Amp, Jules + 14 more
CLAUDE.mdClaude Code
.cursorrulesCursor
.github/copilot-instructions.mdGitHub Copilot
.windsurfrulesWindsurf (Codeium)
GEMINI.mdGoogle Gemini CLI
.continue/rules/00-plansync.mdContinue.dev

Files are written to .plansync/context/<username>/ and use <!-- plansync --> / <!-- end-plansync --> markers. The directory is gitignored so each team member gets their own view. To use, tell your agent to read .plansync/context/<your-username>/ after running plansync sync.

The root AGENTS.md (written by plansync init) always contains the project planning instructions for the admin's agent — it is not overwritten by sync.

Adding more agents: Drop a template file into .plansync/templates/ and it will be rendered for every user at sync time. The filename becomes the output path (e.g., .plansync/templates/.my-agent.md.tmpl → .plansync/context/<user>/.my-agent.md). All templates receive {{username}}, {{userTaskCount}}, {{taskList}}, {{projectTitle}}, {{projectDescription}}, and {{scopeGlobs}} variables.
