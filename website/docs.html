Commands

CommandUse caseDetails
plansync init Set up PlanSync on a repo Authenticates with GitHub via device flow (press Enter, browser opens with 5s countdown) or by pasting a PAT. Saves credentials to .plansync/config.json, adds it to .gitignore (and untracks if previously committed). Copies CI workflows (scope-check.yml, silence-detection.yml, publish.yml) and a post-merge git hook. Writes AGENTS.md with planning instructions for your coding agent.
plansync delegate Push a plan to GitHub Reads .plansync/plan.json (written by your agent). Auto-distributes tasks round-robin to repo collaborators or all to you. Shows the assignment table for review — you can reassign individual tasks before approving. Creates: GitHub Issues per task with scope + acceptance criteria, a Project v2 board, scope manifests (.plansync/scopes/*.json), context files for every agent tool, and PROJECT_PLAN.md. Commits all changes and pushes.
plansync status Check plan + live Issue states Prints every task with its plan status (ready / in progress / done / blocked), GitHub Issue number and state (open / closed), assignee, scope globs, and any blocking dependencies. Shows a summary count at the end.
plansync sync [--user <name>] Lock files to your assigned scope Makes files matching your task scopes writable (chmod 644), everything else read-only (chmod 444). Username resolved via --user flag, PLANSYNC_USER env var, or auto-detected from GitHub token. Validates that you're a real assignee in the plan. Use --reset to restore all files to writable. Skips .git/, node_modules/, and .plansync/.
plansync plan <description> LLM-assisted plan drafting (optional) Requires ANTHROPIC_API_KEY. Sends your project description to an LLM which returns tasks with scope globs, dependencies, and acceptance criteria. Supports an interactive loop: approve, edit with feedback (regenerates), or cancel. Saves the result to .plansync/plan.json.
plansync clean Uninstall PlanSync from a project Removes .plansync/ directory, .github/workflows/publish.yml, scope-check.yml, silence-detection.yml, the post-merge git hook, PROJECT_PLAN.md, and strips <!-- plansync --> blocks from all context files. Removes .plansync/config.json from .gitignore. After clean, run plansync init to start fresh.

Architecture

PlanSync uses four layers:

  1. Plan generation — After plansync init, your coding agent reads AGENTS.md and writes .plansync/plan.json with tasks, scope, and dependencies. You review, revise, approve.
  2. GitHub synchronization — The delegate command writes the plan to GitHub Issues, a Project board, scope manifests, and per-agent context files.
  3. Scope enforcement — The sync command walks the repo and applies filesystem permissions. Only your assigned files are writable.
  4. CI monitoring — A GitHub Action checks every PR for scope violations. A scheduled action flags stalled tasks.

Key design decisions:

  • GitHub is the source of truth — plans, manifests, and context files are plain files committed to the repo
  • No server — PlanSync is a CLI tool and a set of CI actions. Nothing to host.
  • No API gate — Scope enforcement uses chmod. Any tool respects it.
  • CI is the backstop — local permissions prevent accidents; the GitHub Action prevents deliberate scope drift.

Context files

PlanSync generates a context file per agent tool per task, compatible with 28+ coding agents:

FileAgents
AGENTS.mdOpenCode, Codex CLI, Cursor, GitHub Copilot, VS Code AI, Devin, Aider, Zed, Warp, JetBrains Junie, Gemini CLI, Goose, Factory, Amp, Jules + 14 more
CLAUDE.mdClaude Code
.cursorrulesCursor
.github/copilot-instructions.mdGitHub Copilot
.windsurfrulesWindsurf (Codeium)
GEMINI.mdGoogle Gemini CLI
.continue/rules/00-plansync.mdContinue.dev

Each file tells the agent its task description, scope globs, acceptance criteria, and dependencies. The agent can read the full project but is instructed to only write within its scope.

All context files use <!-- plansync --> / <!-- end-plansync --> markers. If you already have custom instructions in these files (e.g. coding conventions, testing preferences), PlanSync appends to or updates its section in-place — your content is never overwritten.

Adding more agents: Drop a template file into .plansync/templates/ and it will be rendered at delegate time. The filename becomes the output path (e.g., .plansync/templates/.my-agent.md.tmpl → .my-agent.md). All templates receive projectTitle, taskDescription, scopeList, scopeGlobs, acceptanceList, and dependencyList variables.
